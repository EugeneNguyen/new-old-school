# when the user comment to an item, add new comment and re-run the whole pipeline

## Analysis

### Scope

**In scope:**

- When a **user** adds a comment to a workflow item (via the dashboard UI or API), the system should automatically restart the item's pipeline from the first stage.
- The newly added comment (and all prior comments) must be preserved across the restart — `restartItem()` already preserves comments.
- The item's `index.md` content generated by prior stages should be truncated (as `restartItem()` already does at the first `## Analysis` heading), so stages re-run against the original request plus the new user feedback.
- The restarted pipeline should incorporate the comment as additional context available to stage agents (comments are already visible in `meta.yml` which agents can read).

**Out of scope:**

- Comments authored by `agent` or `runtime` must NOT trigger a re-run — only `user`-authored comments should. Without this guard, the system would enter an infinite loop: agent finishes stage → runtime posts summary comment → comment triggers restart → repeat.
- Editing or deleting existing comments should not trigger a re-run.
- This requirement does not cover a "selective stage re-run" (e.g., re-run from a specific stage). It is a full pipeline restart.

### Feasibility

**Technical viability: High.** All the building blocks already exist:

1. **Comment append** — `POST /api/workflows/[id]/items/[itemId]/comments` calls `appendItemComment()` in `lib/workflow-store.ts`. The `author` field is already passed through.
2. **Pipeline restart** — `restartItem()` in `lib/workflow-store.ts` resets stage to the first stage, clears sessions, truncates `index.md`, and sets status to `Todo`. The `POST .../restart` route then calls `triggerStagePipeline()` to kick off stage 1.
3. **Stage trigger** — `triggerStagePipeline()` handles session creation and Claude CLI subprocess spawning.

**Implementation approach:** The most straightforward path is to add restart logic to the comment POST route handler (`app/api/workflows/[id]/items/[itemId]/comments/route.ts`). After `appendItemComment()` succeeds and the author is `"user"`, call `restartItem()` + `triggerStagePipeline()`. This mirrors what the existing `POST .../restart` route does.

**Risks:**

- **Race condition with in-progress sessions.** If a stage agent is currently running when the user comments, `restartItem()` clears sessions from `meta.yml` but does not kill the running Claude CLI subprocess. The orphaned subprocess may continue writing to `index.md` or `meta.yml`, conflicting with the newly started stage 1 session. The existing `restartItem()` implementation has this same gap — it is not unique to this feature, but this feature makes it more likely to occur since users may comment while a stage is actively running.
- **Rapid comment spam.** Multiple comments in quick succession would each trigger a full restart. Only the last restart matters; earlier ones waste resources. A debounce or "restart-pending" flag could mitigate this but adds complexity.

### Dependencies

- **`lib/workflow-store.ts`** — `appendItemComment()`, `restartItem()`, `updateItemMeta()`.
- **`lib/auto-advance.ts`** — `triggerStagePipeline()` for kicking off the first stage after restart.
- **`app/api/workflows/[id]/items/[itemId]/comments/route.ts`** — the POST handler where the trigger logic will be added.
- **`lib/auto-advance-sweeper.ts`** — `completeSessionIfFinished()` should handle the orphaned session gracefully (it will eventually time out via idle detection, but may conflict in the interim).
- **Dashboard UI** — the comments panel already sets `author: "user"` when posting. No UI changes expected unless we want to show a "Pipeline restarting..." indicator.
- **Activity log** — `restartItem()` does not currently log an activity entry. Consider whether the restart-via-comment should produce an `item-restarted` activity log entry for auditability.

### Open Questions

1. **Should an in-progress session be killed before restarting?** Currently `restartItem()` does not terminate running Claude CLI subprocesses. If the user comments during an active stage, the old session will keep running alongside the new one. Should we add session termination logic, or accept the current behavior where the orphaned session eventually goes idle and is ignored?
2. **Should there be a way to comment without triggering a re-run?** Some comments may be purely informational (e.g., "FYI, this is related to REQ-00042"). Should there be an opt-out mechanism (e.g., a `triggerRerun: false` flag on the comment API), or should all user comments always trigger a restart?
3. **Debounce behavior for rapid comments?** If a user posts 3 comments in 10 seconds, should each one restart the pipeline independently, or should there be a short debounce window (e.g., 5 seconds) that batches comments into a single restart?
4. **Activity log entry?** Should the restart-via-comment produce a distinct activity log event (e.g., `pipeline-restarted-by-comment`) so operators can distinguish user-triggered restarts from manual restarts via the restart API?

## Specification

### User Stories

1. **As an operator**, I want my comment on a workflow item to automatically restart its pipeline from the first stage, so that I can provide feedback or corrections without manually clicking restart.
2. **As an operator**, I want only my own comments (author = `"user"`) to trigger a restart, so that agent and runtime summary comments do not cause infinite restart loops.
3. **As an operator**, I want the restart triggered by my comment to preserve all existing comments (including the one I just posted), so that stage agents can read my feedback when they re-run.

### Acceptance Criteria

1. **Given** a workflow item at any stage and status, **when** a `POST /api/workflows/{id}/items/{itemId}/comments` request is made with `author: "user"`, **then** the comment is appended and the item's pipeline is restarted from the first stage (status reset to `Todo`, `index.md` truncated at `## Analysis`, sessions cleared).
2. **Given** a comment POST request with `author: "agent"` or `author: "runtime"` (or author omitted, which defaults to `"agent"`), **when** the comment is appended, **then** no pipeline restart occurs.
3. **Given** a successful user-comment-triggered restart, **then** a `triggerStagePipeline()` call is made for the item, starting stage 1 execution.
4. **Given** a successful user-comment-triggered restart, **then** an activity log entry of type `restart` is appended with `actor: "comment"` so operators can distinguish comment-triggered restarts from manual UI restarts.
5. **Given** a successful user-comment-triggered restart, **then** all comments (including the newly added one) are preserved in `meta.yml`.
6. **Given** the workflow has zero configured stages, **when** a user comment is posted, **then** the comment is still appended successfully but no restart is attempted, and no error is returned to the caller.
7. **Given** `restartItem()` returns `null` (item not found during restart), **when** a user comment is posted, **then** the comment has already been appended; the response still returns `201` with the comment, and the restart failure is logged to `console.error` but does not block the response.
8. **Given** the comment POST response, **then** the response shape remains unchanged (`{ ok: true, comment: Comment }` with status `201`) regardless of whether a restart was triggered.

### Technical Constraints

1. **File**: `app/api/workflows/[id]/items/[itemId]/comments/route.ts` — the POST handler is the sole modification point.
2. **Imports to add**: `restartItem`, `readStages` from `@/lib/workflow-store`; `triggerStagePipeline` from `@/lib/stage-pipeline`; `appendActivity`, `ActivityActor` from `@/lib/activity-log`.
3. **Restart call**: Use `restartItem(id, itemId)` followed by `triggerStagePipeline(id, itemId)`, mirroring the existing `POST .../restart` route handler pattern (see `app/api/workflows/[id]/items/[itemId]/restart/route.ts`).
4. **Author guard**: The restart logic MUST only execute when `author === "user"`. The existing default for omitted author is `"agent"`, which must NOT trigger a restart.
5. **Fire-and-forget**: Both `restartItem()` and `triggerStagePipeline()` should be called with `void` (fire-and-forget) after the comment is successfully appended, so the comment POST response is not delayed by restart/pipeline operations.
6. **Activity log**: Use `appendActivity()` with `type: 'restart'` and `actor: 'comment' as ActivityActor` to distinguish from UI-initiated restarts (which use `actor: 'ui'`).
7. **No debounce**: V1 does not implement debounce for rapid comments. Each user comment triggers an independent restart. This is acceptable because the existing `restartItem()` is idempotent and the heartbeat sweeper will clean up orphaned sessions.
8. **No session termination**: V1 does not kill in-progress Claude CLI subprocesses before restarting. This matches the existing `restartItem()` behavior. The orphaned session will eventually go idle and be ignored by the sweeper.
9. **No UI changes**: The dashboard comment panel already sets `author: "user"`. No frontend modifications are required.
10. **Data shapes**: The Comment object shape (per `types/workflow.ts`) is unchanged. The `meta.yml` schema is unchanged.

### Out of Scope

1. **Debounce / batching** — Multiple rapid user comments each trigger an independent restart. A debounce mechanism may be added in a future requirement if this proves problematic.
2. **Session termination** — Killing in-progress Claude CLI subprocesses before restart. This is a pre-existing gap in `restartItem()` and should be addressed as a separate requirement if needed.
3. **Opt-out flag** — A `triggerRerun: false` option on the comment API to post informational comments without triggering a restart. Can be added later if operators need it.
4. **Selective stage re-run** — Restarting from a specific stage rather than the first stage.
5. **UI indicator** — Showing a "Pipeline restarting..." toast or indicator in the dashboard when a comment triggers a restart.
6. **Edit/delete comment triggers** — Only new comment creation (POST) triggers a restart; PATCH and DELETE on comments do not.

## Implementation Notes

### Changes Made

1. **`app/api/workflows/[id]/items/[itemId]/comments/route.ts`** — Added restart logic after `appendItemComment()` succeeds:
   - Imported `readStages`, `restartItem` from `@/lib/workflow-store`, `triggerStagePipeline` from `@/lib/stage-pipeline`, and `appendActivity`, `ActivityActor` from `@/lib/activity-log`.
   - After comment is appended, if `author === "user"`, the code checks `stages.length > 0` (AC6: no restart when zero stages).
   - If stages exist, logs a `restart` activity entry with `actor: 'comment'` (AC4).
   - Calls `restartItem(id, itemId)` and, if it returns non-null, calls `triggerStagePipeline(id, itemId)` (AC3).
   - If `restartItem()` returns null, logs to `console.error` without blocking the response (AC7).
   - Both restart + pipeline trigger use `void` fire-and-forget to keep response fast (constraint 5).
   - Response shape `{ ok: true, comment }` is unchanged (AC8).

2. **`lib/activity-log.ts`** — Added `'comment'` to `ActivityActor` type union to support the new actor value.
