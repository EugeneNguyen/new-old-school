# REQ-016: Stage Prompt Pipeline: Auto-dispatch Todo Items to Agent Session

## Description
When a workflow item arrives at a stage (via creation or drag/drop) and (a) that stage has a non-empty `prompt` configured in `stages.yaml` and (b) the item's `status` is `Todo`, the server sends the stage prompt together with the item's title and body to an agent in a new session via an adapter layer. The resulting `sessionId` is persisted in the item's `meta.yml` so the session can be resumed later. The adapter is pluggable; the default implementation spawns `claude` via the same mechanism used by `app/api/claude/route.ts`.

## Acceptance Criteria
- [x] New `lib/agent-adapter.ts` exports an `AgentAdapter` interface with `startSession({ prompt, cwd }): Promise<{ sessionId: string; adapter: string }>` and a `getDefaultAdapter()` helper.
- [x] Default `ClaudeAdapter` spawns `claude -p --output-format stream-json --verbose --dangerously-skip-permissions`, writes the stream to `.claude/sessions/<id>.txt` (same format as the existing Claude chat), and resolves with the first `session_id` emitted. Subsequent stream events keep writing until the process exits (fire-and-forget).
- [x] `WorkflowItem.sessions?: ItemSession[]` type added (`{ stage, adapter, sessionId, startedAt }`).
- [x] `lib/workflow-store.ts` reads/writes a `sessions` array in `meta.yml` alongside existing fields. `updateItemMeta` exposes a new `appendSession(workflowId, itemId, entry)` helper used by the pipeline.
- [x] New `lib/stage-pipeline.ts` exports `triggerStagePipeline(workflowId, itemId)` that: loads the item + stage, checks `stage.prompt` is non-empty and `item.status === 'Todo'`, builds the full prompt, starts a session via the adapter, and appends the session to `meta.yml`. Errors are caught and logged — they do not throw.
- [x] `POST /api/workflows/[id]/items` calls `triggerStagePipeline` after creating the item (awaits the `sessionId` resolution but swallows errors).
- [x] `PATCH /api/workflows/[id]/items/[itemId]` calls `triggerStagePipeline` after the stage has changed (only when the `stage` field was in the patch and the update succeeded).
- [x] Pipeline is a no-op when the stage has no prompt or the item's status is not `Todo`.
- [x] Re-entering a stage creates a new session entry (history is preserved; nothing is overwritten).
- [x] `readItem`/`readItems` return the `sessions` array so the frontend can display session metadata later.
- [x] `npm run build` succeeds.

## Technical Constraints
- The adapter spawn must be non-blocking on the HTTP request: await only the first stream line that carries `session_id`, then detach and let the child continue writing. A hard timeout (e.g. 10s) rejects if no `session_id` is observed.
- Use the existing `.claude/sessions/<id>.txt` log format (one JSON event per line, `user_prompt` event first) so the session is replayable via the existing `/api/claude/sessions/[id]` endpoints.
- Full prompt format: `${stage.prompt}\n\n# ${item.title}\n\n${item.body ?? ''}`.
- Follow REQ-012 file-first persistence conventions; session metadata lives in `meta.yml`.
- No UI changes in this requirement — frontend surfacing of the session is a follow-up.
