# REQ-011: Workflow Engine

## Description
A multi-step execution engine that lets users define, trigger, and monitor workflows within the nos platform. Workflows are declarative JSON configs consisting of ordered steps — each step can run a shell command, make an API call, or send a Claude prompt. The engine tracks execution state per step and per run, streams real-time status via SSE, and surfaces results in a dedicated dashboard page.

## Acceptance Criteria
- [ ] Workflow definitions stored as JSON configs in `config/workflows.json`
- [ ] Each workflow has a name, description, and ordered list of steps
- [ ] Each step has a type (`shell`, `api`, `claude`), inputs, and expected outputs
- [ ] REST API: CRUD workflows (`/api/workflows`)
- [ ] REST API: Trigger a run (`POST /api/workflows/[id]/run`)
- [ ] REST API: Get run status (`GET /api/workflows/runs/[runId]`)
- [ ] Execution engine runs steps sequentially, passing outputs as inputs to next step
- [ ] Per-step state tracking: `pending` → `running` → `completed` | `failed`
- [ ] Per-run state tracking: `pending` → `running` → `completed` | `failed`
- [ ] Failed steps surface error details (stderr, status code, error message)
- [ ] Real-time status updates via SSE during execution
- [ ] Dashboard page at `/dashboard/workflows` lists available workflows
- [ ] Dashboard page shows run history with status badges
- [ ] Users can trigger a workflow run from the UI
- [ ] Users can view live step-by-step progress of an active run
- [ ] Workflow registered as a tool in `config/tools.json`

## Technical Constraints
- Next.js App Router with TypeScript
- shadcn/ui + Tailwind CSS for UI components
- Config-driven via `config/workflows.json` (matching existing `tools.json` / `skills.json` pattern)
- REST API follows existing error handling conventions (`app/api/utils/errors.ts`)
- SSE streaming follows patterns established in Claude terminal (REQ-005/REQ-009)
- Workflow runs stored as JSON files in `.nos/workflow-runs/`
